# OT Meets IT — Episode 3: Prompt to PLC
# PLC AI Development Workflow

## Overview
This workflow guides Claude Code through AI-assisted PLC program development using a
structured, repeatable process. The agent will generate a Software Functional Requirements
(SFR) document, produce IEC 61131-3 Structured Text code, validate it, manage version
control, and write the code into a CodeSys project file using the CodeSys MCP.

The user will provide a **topic** at the start of the session. The agent derives all file
names and program logic from that topic and handles everything through to a buildable
project file.

---

## Directory Structure

The agent must verify the following structure exists at session start:

```
/Prompt-to-PLC/
├── CLAUDE.md                           ← this file (auto-read by Claude Code)
├── .gitignore
├── README.md
├── /sfr/
│   └── {topic}_SFR.md                 ← generated by agent in Phase 2
├── /code/
│   ├── {topic}_declaration.txt        ← generated by agent in Phase 3
│   └── {topic}_program.txt            ← generated by agent in Phase 3
└── /project/
    └── Template_750_8111.project      ← committed to repo, never modified directly
```

If any folder is missing, create it before proceeding.

---

## File Naming Convention

All file names are derived from the topic provided by the user, lowercase with underscores:

| Artifact | Path |
|---|---|
| SFR document | `/sfr/{topic}_SFR.md` |
| Declaration file | `/code/{topic}_declaration.txt` |
| Program file | `/code/{topic}_program.txt` |
| Working project | `/project/{topic}_v1.project` (gitignored) |

## Workflow Steps

---

### PHASE 0 — Session Initialization

1. Read this file completely before taking any action
2. Confirm the directory structure exists; create any missing folders
3. Ask the user for the **topic** if not already provided
4. Confirm `Template_750_8111.project` exists in `/project/`
5. Confirm Git is initialized and `.gitignore` is in place
6. **Verify the CodeSys MCP is the required (forked) build.** This workflow depends on the
   forked MCP's `get_pou_code` tool (for reading a POU back) and its structured
   `compile_project` build results. Checking that the **`get_pou_code` tool is present in the
   CodeSys MCP toolset** is sufficient — it is unique to the fork, so no build probe is needed.

   If `get_pou_code` is not present, **stop** — the connected MCP is the stock build, not the
   forked version this workflow requires. Tell the user to install and point Claude Code at
   the forked CodeSys MCP (https://github.com/Moses-Ling/codesys-mcp-toolkit.git), and do not
   proceed to Phase 1.

---

### PHASE 1 — Plan

1. Enter **Plan Mode**
2. Ask the user for a **User Story** — a plain-language description, in their own words, of
   what they want the program to do (e.g. "As an operator, I want the LED to blink every
   second so I know the PLC is running"). The topic name provided in Phase 0 is only a label
   for file naming — it is never a substitute for the User Story. Do not generate any
   clarifying questions or assumptions about the intended behavior until the user has given
   this story.
3. Based on the user's story, ask clarifying questions to fill in the gaps needed for
   an SFR:
   - What inputs and outputs will the PLC program need?
   - What sequences, timers, or states are involved?
   - What safety conditions and edge cases should the SFR capture?
   - What IEC 61131-3 data types and function blocks are appropriate?
4. Where more than one reasonable implementation approach exists, present the options to the
   user and let them choose — do not silently pick one
5. Continue this back-and-forth until there is enough information to build the skeleton and
   body of a draft SFR
6. Produce a written plan summary — do not write any files yet
7. Present the plan to the user and wait for confirmation before proceeding

---

### PHASE 2 — Write the SFR

1. Generate a **draft** SFR document from the plan agreed in Phase 1 and write it to
   `/sfr/{topic}_SFR.md`
2. The SFR must include the following sections:

   - **Purpose** — what the program does and why
   - **Inputs** — variable names, data types, descriptions
   - **Outputs** — variable names, data types, descriptions
   - **Internal Variables** — timers, counters, flags, state variables
   - **Functional Requirements** — numbered, testable requirement statements
   - **Sequence of Operation** — step-by-step logic description
   - **Edge Cases & Safety** — what the program must handle gracefully

3. Present the draft SFR to the user and work with them iteratively — incorporate feedback
   and revise the file — until it converges on a final SFR
4. Wait for the user's explicit approval of the final SFR before proceeding to Phase 3

---

### PHASE 3 — Generate Code

Based on the approved SFR, generate two separate files:

**`/code/{topic}_declaration.txt`**
- All VAR declarations — inputs, outputs, and internal variables
- Correct IEC 61131-3 data types matching the SFR exactly
- Inline comments for every variable explaining its purpose

**`/code/{topic}_program.txt`**
- Full Structured Text (ST) program body
- Logic must satisfy every numbered requirement in the SFR
- Inline comments explaining each logical block
- No undeclared variables

Write both files to the `/code/` directory before proceeding.

---

### PHASE 4 — Self-Validation

Before touching the CodeSys project file, perform a structured validation against the SFR.

**Checks to perform:**

1. **SFR Coverage** — every numbered functional requirement is addressed in the program
2. **Declaration Consistency** — every variable used in the program is declared; every declared variable is used
3. **Type Consistency** — data types are consistent between the declaration and program files
4. **Logic Review** — check for common PLC coding errors:
   - Uninitialized timers or counters
   - Missing ELSE branches on critical IF statements
   - Unreachable code paths
   - Output coil conflicts (same output written in multiple places)

**Output:**
- Produce a Validation Report listing each check and its pass/fail result
- If any issues are found, correct the text files before proceeding
- Present the Validation Report to the user
- Wait for user approval before proceeding to Phase 5

---

### PHASE 5 — Git Commit (Pre-Build)

Commit the SFR and generated code files **locally only**:

```bash
git add sfr/ code/
git commit -m "feat({topic}): add SFR and generated ST code"
```

Do not stage or commit anything in `/project/`. Do not push — this workflow stays local-only
unless the user has explicitly set up a remote and asked for a push (see Phase 10).

---

### PHASE 6 — CodeSys MCP Write

1. Copy `/project/Template_750_8111.project` → `/project/{topic}_v1.project`
2. Use the CodeSys MCP to open `/project/{topic}_v1.project`
3. Write the contents of `{topic}_declaration.txt` into the **Declaration section** of the main POU
4. Write the contents of `{topic}_program.txt` into the **Implementation section** of the main POU
5. Verify the write: read the POU back with `get_pou_code` and confirm both sections match
   the text files in `/code/` — if they do not match, rewrite before proceeding
6. Save the project file via the CodeSys MCP

---

### PHASE 7 — Build

The CodeSys MCP's `compile_project` tool returns the build result directly: a
`Build PASSED` / `Build FAILED` status, `BUILD_ERROR:` / `BUILD_WARNING:` lines, and a
`BUILD_RESULT: N error(s), M warning(s)` summary. No manual relay from the user is needed.

1. Use the CodeSys MCP to build `{topic}_v1.project`
2. Parse the returned `BUILD_RESULT` line for the error and warning counts
3. Report any warnings to the user even on a passing build — warnings do not block
   progress, but the user must be made aware of them

- **0 errors** → proceed to Phase 9
- **1 or more errors** → proceed to Phase 8

---

### PHASE 8 — Error Correction Loop

Repeat until the build passes or 3 iterations are reached:

1. Parse the `BUILD_ERROR:` lines returned by `compile_project`
2. Read the failing POU back with `get_pou_code` — error positions such as
   `Line 5, Column 1 (Impl)` are line numbers within the Declaration or Implementation
   section as returned by that tool
3. Identify which file requires correction — declaration, program, or both
4. Apply the fix to the appropriate text file(s) in `/code/`
5. Save the corrected file(s) locally
6. Use the CodeSys MCP to write the corrected content into `{topic}_v1.project`
7. Rebuild via `compile_project`, evaluate the returned result, and repeat if necessary

**If 3 iterations are reached without a passing build:**
- Document all remaining errors clearly
- Present to the user with specific recommended manual fixes
- Stop the loop and await user guidance before continuing

---

### PHASE 9 — Final Git Commit (Post-Build)

After a passing build, commit any corrected text files **locally only** (no push):

```bash
# If corrections were made during the error loop:
git add code/
git commit -m "fix({topic}): corrections from CodeSys build validation"

# If the first build passed cleanly:
git commit -m "build({topic}): code validated, build passing — no corrections needed"
```

---

### PHASE 10 — Handoff to User

Present a handoff summary containing:

1. **Project file location** — `/project/{topic}_v1.project`
2. **Program summary** — what the program does (derived from the SFR purpose)
3. **I/O mapping table** — list all input and output variables that require hardware address mapping
4. **Build evidence** — the final `BUILD_RESULT` line returned by `compile_project`, plus any
   warnings that were reported
5. **Verification checkpoint** — ask the user to verify the SFR, code, and build result, and
   confirm everything is committed locally (`git log`/`git status` should be clean)
6. **Remote reminder** — this repo is local-only; nothing has been pushed anywhere. If the
   user wants to back it up or collaborate, remind them they can add a remote and push
   whenever they're ready:
   ```bash
   git remote add origin <new-repo-url>
   git push -u origin main
   ```
7. **Next steps for the user:**
   - Open `{topic}_v1.project` in the CodeSys IDE
   - Navigate to the Device Tree
   - Map each I/O variable to its corresponding hardware address
   - Rebuild the project in the IDE
   - Download to the PLC and run

## Rules

1. **Never modify `Template_750_8111.project`** — always copy it first; the template is the baseline
2. **Text files are the source of truth** — the project file is a build artifact, not source code
3. **Always run Phase 4 validation before the CodeSys MCP write** — it is not optional
4. **One topic per session** — start a fresh session for each new PLC program
5. **Never commit `.project` files** — the `.gitignore` enforces this; do not override it
6. **When in doubt, ask the user** — do not invent requirements or assume hardware details
7. **Never push automatically** — every commit in this workflow (Phase 5, Phase 9) is local-only;
   only push if the user has explicitly set up a remote and asked for it
8. **Requires the forked CodeSys MCP** — Phases 6–8 depend on `get_pou_code` and on
   `compile_project` returning `BUILD_RESULT`. The forked MCP is at
   https://github.com/Moses-Ling/codesys-mcp-toolkit.git. Run the Phase 0 preflight check
   first; if the MCP lacks these, stop rather than proceeding with a build the agent cannot read
